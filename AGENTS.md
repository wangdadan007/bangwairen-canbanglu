## 1. 项目一句话

《榜外人：残榜录》是一款封神题材单机牌组构筑肉鸽。玩家扮演封神榜外的执簿者，在战斗中破敌之形、问敌之名、正名归册，并通过裁定逐步改写一卷破裂的封神榜。

它不是：

- 传统封神神将收集。
- 商周阵营战役复刻。
- 敌魂变卡。
- Boss 饶恕助战。
- “封神皮肤的普通爬塔卡牌”。

核心体验必须始终围绕：

> 破形、问名、正名、归册、裁定，改写残榜。

## 2. 当前阶段

当前真实开发阶段、已完成任务、下一步入口和本轮任务边界，以 `docs/tasks/PROJECT_PLAN.md` 为准。

`AGENTS.md` 只保留稳定规则和读取入口，不再记录“当前阶段只做 / 暂时不做”的流水清单，避免 AI 在后续会话中按过期阶段判断任务范围。

不要一次性实现完整游戏。每次只完成用户当前要求的模块。

如果 `PROJECT_PLAN.md`、PRD 或用户最新指令之间出现阶段或任务范围冲突，先按本文件第 0 节的事实源优先级判断；仍不清楚时，把问题记录到 `docs/tasks/OPEN_QUESTIONS.md`，不要擅自改核心方向。

## 3. 技术方向

计划技术栈：

- TypeScript
- Vite
- React
- Phaser 4.1.x
- Vitest
- 桌面版本先预留 Electron 方向，后续可再评估 Tauri

第一阶段优先让浏览器内核心战斗 MVP 可运行、可测试、可调试。不要先做桌面打包。

不要擅自新增生产依赖。确实需要新增依赖时，先说明：

- 为什么需要。
- 是否能用现有方案替代。
- 体积、维护、构建和授权影响。
- 具体安装版本。

## 4. 目录约定

代码目录以 `docs/prd/PRD-architecture.md` 的最新结构为准：

```text
src/
  core/                  # 纯逻辑层，不依赖 React / Phaser
    battle/
    rng/
    log/
  data/                  # 游戏数据与 localization
  game/                  # Phaser 场景与游戏渲染
  ui/                    # React 外层 UI、页面、调试面板
  types/                 # 共享 TypeScript 类型
  tests/                 # 核心规则测试
```

若旧文档出现 `src/game/core` 之类路径，启动新任务前以 `PRD-architecture.md` 为准统一，不要两套目录并行。

## 5. 架构硬约束

- `src/core` 是纯战斗逻辑层，不得 import React、Phaser、DOM、浏览器存储或渲染对象。
- React 负责外层 UI、菜单、HUD、调试面板；Phaser 负责战斗场景表现；二者都不承载核心规则。
- 卡牌、敌人、遭遇、法宝、事件、裁定尽量数据驱动。
- 每张卡、每个敌人、每个遭遇、每件法宝、每个事件都必须有唯一 ID。
- 中文显示文案放入数据或 localization，不要硬编码在核心逻辑里。
- 所有关键状态变化必须进入 `actionLog`，方便测试、回放和调试。
- 每个核心规则都要有测试；规则越底层，测试越优先。
- 数据文件新增卡牌或敌人时，不应需要修改 React 组件或 Phaser Scene。

## 6. 玩法术语红线

必须使用本项目术语：

| 错误表达 | 正确表达 |
|---|---|
| 血量 / HP | 形 |
| 造成伤害 | 破形 |
| 格挡 | 封势 |
| 攻击意图 | 来势 |
| 特殊行动 / debuff 行动 | 异动 |
| 击杀奖励 | 伏诛 / 归册奖励 |

核心规则：

- 敌人的生命 / 战斗实体叫“形”。
- 玩家削减敌人的形叫“破形”。
- 形归零是唯一基础胜利条件。
- 未正名时形归零：伏诛，普通奖励。
- 正名后形归零：归册，高质量奖励，可能进入裁定。
- 问名用于揭示敌人的名；全部名格揭示后触发正名。
- 正名触发名破，削减敌人形，并可能解除敌人机制。
- 裁定只保留登簿、朱批、削籍。
- 来势和异动必须分离。封势只能处理来势，不能处理异动。
- 法宝不是卡牌本体；法宝在牌组外作为器物存在，可以生成临时牌，但法宝本体不进入牌堆。

## 7. 明确禁止实现

除非用户以后明确改需求并更新 PRD / changelog，不要实现：

- 留魂。
- 留名。
- 敌魂卡。
- Boss 饶恕后助战。
- 神将收集或抽卡养成。
- 商周阵营战役复刻。
- 把法宝本体洗入抽牌堆。
- 把封势设计成万能防御。
- 把三坛做成普通延迟破形 / 延迟封势。
- 在宣传或 README 中写“封神版杀戮尖塔”“类杀戮尖塔”“AI 生成游戏”。

## 8. 必须停下来问用户

以下情况不能自行拍板：

1. 新增生产依赖。
2. 改技术栈或打包方向，例如从 Electron 改 Tauri。
3. 改核心机制定义，例如形、名、正名、伏诛 / 归册、裁定三类。
4. 改当前阶段范围，例如从核心战斗 MVP 直接扩到完整第一章。
5. 改内容规模，例如卡牌、敌人、法宝、事件数量目标。
6. 改目录结构并影响已有任务或文档。
7. 引入第三方素材、字体、音频、美术资源且授权不清。
8. 删除或大幅重写 PRD / 设计文档。

## 9. 开发纪律

- 一次只做一个小功能，不主动扩大范围。
- 修改前先说明影响范围和涉及文件。
- 优先采用仓库已有模式与 PRD 约定，不另起一套抽象。
- 不把玩法规则写死进 React 组件或 Phaser Scene。
- 不编造测试结果。能运行的命令就运行；命令不存在或失败时直接说明。
- 如果同一个问题连续修两次仍失败且没有新线索，停下来说明完整报错、已尝试方案和当前文件状态。
- 不要提交或保留开发密钥、账号、API token。
- 第三方素材必须记录来源、授权、用途和是否可商用；授权不清不得进入正式构建。

## 10. 任务收尾格式

每次开发任务完成后，回复中至少包含：

- 本次目标。
- 修改文件列表。
- 关键实现说明。
- 已完成内容。
- 未完成风险。
- 已运行的命令与结果。
- 下一步建议。

每次完成一个编号任务或阶段性开发任务后，必须同步更新 `docs/tasks/PROJECT_PLAN.md`：

- 将已完成任务的状态改为已完成。
- 记录本次完成内容、验证基线和下一步建议。
- 增加模板沉淀检查：本阶段是否产生可模板化模块？是否需要记录到 `docs/templates/TEMPLATE_EXTRACTION_NOTES.md`？
- 如果任务范围、核心机制、技术栈或阶段规划发生变化，再同步更新 `docs/prd/PRD-changelog.md`。

如果只是文档整理或很小的改动，可以压缩格式，但不能省略测试 / 命令是否运行的真实说明。
