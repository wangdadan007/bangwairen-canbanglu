# 《榜外人：残榜录》首批 Codex 开发任务

版本 v0.1 · 2026-05-16

当前目标：不是完整开发第一章，而是先完成“核心战斗 MVP”。

## 总体启动顺序

1. 建 GitHub 仓库与项目骨架。
2. 放入 `AGENTS.md` 项目开发指令。
3. 建 TypeScript + Vite + React + Phaser 基础工程。
4. 建纯逻辑层，不依赖 UI。
5. 建数据文件和类型。
6. 实现最小战斗循环。
7. 实现破形、问名、正名、伏诛 / 归册。
8. 做最小可视化 UI。
9. 写基础测试。
10. 再进入前三场教学战和奖励系统。

优先级：P0

所属模块：工程基础

背景：项目需要交给 Codex 长期开发，必须先让仓库结构、项目指令、开发脚本和基础技术栈稳定。

目标：建立可运行的 TypeScript + Vite + React + Phaser 项目，并在仓库根目录加入 AGENTS.md。

具体需求：

- 初始化项目。
- 配置 TypeScript。
- 配置 Vite。
- 接入 React。
- 接入 Phaser。
- 创建最小 Phaser 场景并嵌入 React 页面。
- 新增 AGENTS.md，写入项目开发约束。

- `npm install` 成功。
- `npm run dev` 成功启动。
- 浏览器能看到一个最小战斗占位画面。
- 仓库根目录存在 AGENTS.md。

```text
请为《榜外人：残榜录》建立 TypeScript + Vite + React + Phaser 项目骨架。创建基础目录结构，加入 AGENTS.md 和 README.md。先只做一个可运行的空战斗场景，不要实现完整玩法。要求 npm install 和 npm run dev 可运行。
```

## 任务 T01

任务 ID：T01

任务名称：建立核心数据结构与类型

优先级：P0

所属模块：战斗核心

背景：本项目必须数据驱动，不能把卡牌、敌人和效果写死在 UI。

目标：定义 Card、Enemy、CombatState、PlayerState、ActionLog、NameSlot、EnemyIntent 等核心类型。

具体需求：

- 定义 CardDefinition：id、name、cost、type、tags、rulesText、effects。
- 定义 EnemyDefinition：id、name、shapeMax、nameSlots、intents、traits。
- 定义 CombatState：turn、phase、player、enemies、drawPile、hand、discardPile、exhaustPile、log、result。
- 定义 PlayerState：incense、maxIncense、deck、unlocks。
- 定义 ActionLogEntry：type、sourceId、targetId、payload、turn。
- 定义基础 effect 类型：BREAK_SHAPE、ASK_NAME、SEAL_MOMENTUM、DRAW、GAIN_INCENSE。

验收标准：

- 类型文件可通过 TypeScript 编译。
- 无 `any` 滥用。
- 类型能支撑后续 cards.json 和 enemies.json。

交给 Codex 的 prompt：

```text
请在项目中建立核心 TypeScript 类型，覆盖卡牌、敌人、战斗状态、玩家状态、敌人名格、敌人行动意图和战斗日志。不要实现 UI。类型必须支持数据驱动效果。
```

依赖任务：T00

风险：类型过度复杂会拖慢首版开发；先保持可扩展但不要泛化过度。

## 任务 T02

任务 ID：T02

任务名称：建立 cards.json 与 enemies.json 初版数据

优先级：P0

所属模块：数据驱动内容

背景：核心战斗 MVP 需要最小可玩数据。

目标：建立 6 张初始牌和 1 个教学敌人“纸面鬼”。

具体需求：

cards.json 至少包含：

- 朱符：1 香火，破形 4。若目标已有名被揭示，额外破形 1。
- 问名：1 香火，揭示目标 1 格名。若目标无名可问，改为辨势：查看其下一次行动，并破形 1。
- 护案符：1 香火，封势 3。
- 点额：1 香火，破形 2，并问名 1。
- 整卷：0 香火，查看抽牌堆顶 2 张，选择 1 张置底。MVP 可先简化为抽 1 张或记录待实现。
- 静香：0 香火，本回合下一次问名后，获得 1 香火。MVP 可先简化为直接获得 1 香火并消耗。

 enemies.json 至少包含：

- 纸面鬼：形 18，2 格名，简单来势行动。

验收标准：

- 数据文件可被加载。
- 每个条目有唯一 ID。
- 无中文字段名作为逻辑 key，中文只放显示文本。

交给 Codex 的 prompt：

```text
请建立 cards.json 与 enemies.json 初版数据，并写数据加载函数。先放入 6 张初始牌和教学敌人“纸面鬼”。每个对象必须有唯一 ID，逻辑 key 用英文，显示文本可以中文。
```

依赖任务：T01

风险：效果语义如果直接写自然语言，后续无法执行；必须同时有结构化 effects。

## 任务 T03

任务 ID：T03

任务名称：实现核心战斗状态机

优先级：P0

所属模块：战斗核心

背景：需要先跑通一场最小战斗，再谈路线、商店和法宝。

目标：实现抽牌、费用、出牌、弃牌、回合结束、敌人行动、胜负结算。

具体需求：

- 初始化战斗。
- 每回合开始：获得 3 香火，抽 5 张牌。
- 出牌：检查费用，执行效果，移动到弃牌堆或消耗区。
- 回合结束：弃掉手牌，敌人执行简单行动，进入下一回合。
- 抽牌堆为空时洗弃牌堆。
- 支持单敌人战斗。
- 输出 action log。

验收标准：

- 能从开始战斗打到敌人形归零。
- 每个关键动作都有 log。
- 逻辑层不依赖 React 或 Phaser。

交给 Codex 的 prompt：

```text
请实现核心战斗状态机：初始化战斗、回合开始、抽牌、出牌、费用扣除、弃牌、回合结束、敌人简单行动、抽牌堆洗牌、战斗日志。逻辑层不得依赖 React 或 Phaser。
```

依赖任务：T01、T02

风险：如果战斗逻辑写在 UI 中，后续测试和重构会困难。

## 任务 T04

任务 ID：T04

任务名称：实现破形、问名、正名、名破、伏诛 / 归册

优先级：P0

所属模块：核心机制

背景：这是本作最小差异化闭环。

目标：实现“形”和“名”的核心结算。

具体需求：

- 破形 X：降低目标 shapeCurrent。
- 问名 X：揭示目标未揭示 nameSlots。
- 若目标无名可问，问名改为辨势：MVP 先记录 log 并破形 1。
- 全部名格揭示后，目标正名。
- 正名触发名破：普通敌人按最大形 30% 到 35% 削减，MVP 可先取 33%。
- 敌人形归零时，根据是否正名结算为伏诛或归册。

验收标准：

- 未问名打空纸面鬼，结果为伏诛。
- 问名完整后打空纸面鬼，结果为归册。
- 正名触发一次名破，不重复触发。
- action log 能显示问名、正名、名破、伏诛/归册。

交给 Codex 的 prompt：

```text
请实现《榜外人：残榜录》的核心机制：破形、问名、正名、名破、伏诛/归册。敌人形归零是唯一基础胜利条件。正名只影响奖励质量和名破，不是额外胜利条件。
```

依赖任务：T03

风险：正名不能变成第二套胜利条件；形归零必须始终是唯一基础胜利条件。

## 任务 T05

任务 ID：T05

任务名称：最小战斗 UI

优先级：P0

所属模块：UI / 交互

背景：需要尽快看到核心循环是否好玩。

目标：实现一个粗糙但可玩的战斗界面。

具体需求：

- 显示玩家香火。
- 显示手牌。
- 点击手牌后选择目标并出牌；MVP 可默认选择唯一敌人。
- 显示敌人形：当前 / 最大。
- 显示敌人名格：未揭示、已揭示、正名状态。
- 显示战斗日志。
- 显示结束回合按钮。
- 显示战斗结果：伏诛或归册。

验收标准：

- 玩家可以通过 UI 打完一场纸面鬼战斗。
- 能看出问名、正名、名破、伏诛 / 归册。
- UI 文案使用“形”“破形”“问名”“正名”，不出现“血量”“伤害”。

交给 Codex 的 prompt：

```text
请为核心战斗 MVP 实现最小战斗 UI。要求能显示香火、手牌、敌人形、敌人名格、战斗日志、结束回合按钮和战斗结果。MVP 可以只有一个敌人目标。UI 文案必须使用项目术语：形、破形、问名、正名、伏诛、归册。
```

依赖任务：T03、T04

风险：不要过早追求美术表现；先保证规则反馈清楚。

## 任务 T06

任务 ID：T06

任务名称：基础规则测试

优先级：P0

所属模块：测试

背景：本项目规则多，必须尽早建立测试习惯。

- 测试未正名形归零为伏诛。
- 测试正名后形归零为归册。
- 测试出牌费用不足时不能出牌。
- 测试抽牌堆为空时洗弃牌堆。

验收标准：

- `npm test` 可运行。
- 核心规则测试通过。
- 测试不依赖 UI。

交给 Codex 的 prompt：

```text
请为核心战斗 MVP 编写基础单元测试，覆盖破形、问名、正名、名破、伏诛/归册、费用不足、抽牌堆洗牌。测试必须直接调用逻辑层，不依赖 React 或 Phaser。
```

依赖任务：T03、T04

风险：如果没有测试，后续加入朱批、封势、异动、三坛、法宝后很容易互相污染。
