## 1. 文件用途

本文件记录当前开发进度、下一步任务和中期规划，作为 Codex 进入项目时判断“接下来做什么”的任务入口。

它不替代：

- `AGENTS.md`：稳定开发规则、术语红线、协作纪律。
- `docs/prd/`：产品目标、机制事实源、验收标准。
- `docs/tasks/FULL_PROJECT_PLAN.md`：历史完整路线归档指针；不再作为主动维护计划入口。
- `docs/tasks/codex_tasks_v0.1.md`：首批 T00 到 T06 的原始任务说明。
- `docs/prd/PRD-changelog.md`：范围、机制、技术栈等需求变更记录。
- `docs/templates/TEMPLATE_EXTRACTION_NOTES.md`：可复用游戏模板、工具包和商业化抽象的候选沉淀区。

若本文件与 PRD 或用户最新指令冲突，按 `AGENTS.md` 中的事实源优先级处理。

### T07：来势 / 封势 / 异动边界

状态：已完成
优先级：P0
目标：把敌人行动压力做实，并在代码层保证封势不能处理异动。

范围：

- 将来势从日志反馈推进为可被封势处理的战斗状态。
- 实现至少一个异动，例如偷香或塞污卷。
- 实现至少一张专门应对异动的卡牌或效果。
- UI 或调试面板明确区分来势与异动。
- 补测试：封势能处理来势，但不能处理异动。

不做：

- 不做完整敌人池。
- 不做路线、商店、事件。
- 不做完整三坛或法宝。

验收：

- 来势有数值。
- 封势只降低来势。
- 异动能执行。
- 封势不能影响异动。
- actionLog 记录来势、封势、异动执行或反制。

完成记录：

- `EnemyState.incomingForce` 成为当前来势状态，封势会扣减该值。
- 偷香鼠提供偷香异动，未反制时降低下一回合香火。
- 断供符提供专门断异动效果，能阻止偷香。
- UI 显示当前来势、异动预警，并支持在测试战斗中切换纸面鬼 / 偷香鼠。
- 新增 `src/tests/enemyIntent.test.ts` 覆盖来势、封势、异动执行和反制。

### 开发策略决策

建议先做成第一章可以完整游玩，再补正式美术、复杂动效、音频和桌面包装。

理由：

- 当前最大缺口不是系统骨架，而是内容密度、遭遇变化、精英差异、Boss 变体、长期奖励池和平衡闭环。
- 第一章内容和数值在 T31 到 T43 期间还会频繁调整，过早量产正式美术和复杂动效会带来返工。
- 关键玩法反馈不能完全后置。破形、问名、正名、归册、裁定、法宝反噬、Boss 压力等必须先有清楚的占位表现，正式资产可以后补。

优先级顺序：

1. 第一章完整可玩：内容池、精英、Boss、法宝、事件、路线、奖励、商店、休整、结算、存档和基础 QA。
2. 第一章可理解：术语、tooltip、日志、错误态、响应式和长流程数值记录。
3. 第一章可展示：最小表现强化、风格基线、素材授权清单、音频占位和候选版收口。
4. 后置正式量产：正式美术资产、复杂动画、正式音频、Steam / 桌面打包和宣传素材。

## 10. 长期路线总览

本节合并自原 `FULL_PROJECT_PLAN.md`，用于在同一个主动计划入口中保留中长期路线、阶段闸门和覆盖关系。后续任务执行仍以“当前推荐下一步”和编号任务队列为准。

总体目标：

| 层级 | 名称 | 目标 |
|---|---|---|
| 阶段 A | 核心战斗 MVP | 一场战斗可玩、可测试、可复盘，跑通破形、问名、正名、伏诛 / 归册 |
| 阶段 B | 教学纵切 MVP | 连续 3 到 5 场战斗，验证机制解锁、奖励、朱批和最小局内成长 |
| 阶段 C | 第一章公开 Demo | 完整第一章候选版，含路线、商店、休整、事件、精英、Boss、法宝、结算 |
| 阶段 D | 第一章完整可玩硬化 | 手动完整局、全页面回归、数值调平、薄弱经济补强、外部试玩准备和收口 |
| 阶段 E | 第一章三角色扩展 | 3 个原创可选执簿者、角色选择、三套初始差异和基础 QA |

正式版方向暂按 4 个常规章 + 1 个终审短章预留，但当前开发只承诺第一章。第二章必须等第一章三角色扩展是否执行和收口后再单独确认。

核心原则：

- 始终先保证核心规则可运行、可测试、可回放，再扩内容量和表现。
- `src/core` 保持纯逻辑，不依赖 React、Phaser、DOM、浏览器存储或渲染对象。
- 卡牌、敌人、遭遇、法宝、事件、裁定尽量数据驱动。
- 新增卡牌、敌人或遭遇时，不应要求修改 React 组件或 Phaser Scene。
- 所有关键状态变化必须进入 `actionLog`。
- 伏诛路线能通关，归册路线奖励更高；正名不是第二套胜利条件。
- 封势只能处理来势，不能处理异动，这个边界必须在代码层成立。
- 法宝是牌组外器物，法宝本体不进入抽牌堆。
