## 1. 文件用途

本文件记录当前开发进度、下一步任务和中期规划，作为 Codex 进入项目时判断“接下来做什么”的任务入口。

它不替代：

- `AGENTS.md`：稳定开发规则、术语红线、协作纪律。
- `docs/prd/`：产品目标、机制事实源、验收标准。
- `docs/tasks/FULL_PROJECT_PLAN.md`：从核心战斗 MVP 到第一章公开 Demo 的完整版项目路线图。
- `docs/tasks/codex_tasks_v0.1.md`：首批 T00 到 T06 的原始任务说明。
- `docs/prd/PRD-changelog.md`：范围、机制、技术栈等需求变更记录。

若本文件与 PRD 或用户最新指令冲突，按 `AGENTS.md` 中的事实源优先级处理。

## 3. 进度更新规则

每完成一个编号任务或阶段性开发任务，都必须更新本文件。

更新要求：

- 已完成的任务状态改为“已完成”。
- 当前推荐下一步保持唯一且明确。
- 若新增任务，写入目标、范围、不做内容和验收标准。
- 若本次任务改变验证基线，更新“当前验证基线”。
- 若任务范围、核心机制、技术栈或阶段规划发生变化，同时更新 `docs/prd/PRD-changelog.md`。
- 若只是代码实现或测试补齐，没有改变需求范围，不需要更新 changelog。

推荐收尾顺序：

1. 完成代码或文档改动。
2. 运行该任务需要的验证命令。
3. 更新本文件的任务状态、当前下一步和验证基线。
4. 如有需求变化，再更新 `docs/prd/PRD-changelog.md`。
5. 最终回复中说明本文件已更新。

## 5. 阶段 A 收口记录

### T07：来势 / 封势 / 异动边界

状态：已完成
优先级：P0
目标：把敌人行动压力做实，并在代码层保证封势不能处理异动。

范围：

- 将来势从日志反馈推进为可被封势处理的战斗状态。
- 实现至少一个异动，例如偷香或塞污卷。
- 实现至少一张专门应对异动的卡牌或效果。
- UI 或调试面板明确区分来势与异动。
- 补测试：封势能处理来势，但不能处理异动。

不做：

- 不做完整敌人池。
- 不做路线、商店、事件。
- 不做完整三坛或法宝。

验收：

- 来势有数值。
- 封势只降低来势。
- 异动能执行。
- 封势不能影响异动。
- actionLog 记录来势、封势、异动执行或反制。

完成记录：

- `EnemyState.incomingForce` 成为当前来势状态，封势会扣减该值。
- 偷香鼠提供偷香异动，未反制时降低下一回合香火。
- 断供符提供专门断异动效果，能阻止偷香。
- UI 显示当前来势、异动预警，并支持在测试战斗中切换纸面鬼 / 偷香鼠。
- 新增 `src/tests/enemyIntent.test.ts` 覆盖来势、封势、异动执行和反制。

### T08：战斗 UI 补强

状态：已完成
优先级：P0
目标：让玩家在 UI 上清楚理解来势、封势、异动的区别。

范围：

- 显示异动预警。
- 显示封势处理结果。

- 敌方面板新增封势 / 断异动判断，让玩家看到当前可压住来势、异动未处理或专门处理已布置。

## 8. 第一章完成路线

阶段目标：把当前“第一章线性可收束骨架”推进到“第一章可完整游玩、可测试、可调平衡、可作为公开 Demo 候选版”。

### 开发策略决策

建议先做成第一章可以完整游玩，再补正式美术、复杂动效、音频和桌面包装。

理由：

- 当前最大缺口不是系统骨架，而是内容密度、遭遇变化、精英差异、Boss 变体、长期奖励池和平衡闭环。
- 第一章内容和数值在 T31 到 T43 期间还会频繁调整，过早量产正式美术和复杂动效会带来返工。
- 关键玩法反馈不能完全后置。破形、问名、正名、归册、裁定、法宝反噬、Boss 压力等必须先有清楚的占位表现，正式资产可以后补。

优先级顺序：

1. 第一章完整可玩：内容池、精英、Boss、法宝、事件、路线、奖励、商店、休整、结算、存档和基础 QA。
2. 第一章可理解：术语、tooltip、日志、错误态、响应式和长流程数值记录。
3. 第一章可展示：最小表现强化、风格基线、素材授权清单、音频占位和候选版收口。
4. 后置正式量产：正式美术资产、复杂动画、正式音频、Steam / 桌面打包和宣传素材。
