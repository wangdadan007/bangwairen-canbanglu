## v1.5 · 2026-05-20

### 来源

- 用户确认在第一章完整可玩版后补工程质量门禁，要求把 ESLint、覆盖率基线和 Playwright 浏览器冒烟写成 T61；暂时不过度加严的项目写入后续计划。

### 新增

- 新增 T61 工程质量门禁任务。
- 新增 ESLint、Vitest coverage、Playwright 最小浏览器冒烟和统一验证脚本。
- GitHub Actions 增加 lint、coverage baseline 和浏览器 smoke。

### 修改

- 当前验证口径从 `typecheck + test + build` 扩展为日常 `verify`，完整质量检查可运行 `verify:full`。
- 覆盖率本轮只建立基线，不设置硬阈值。
- Playwright 本轮只覆盖标题页、设置页、新局法宝三选一、进入第一战、1280x720 溢出和浏览器 error。

### 原因

- 用户不读代码，需要更稳定、可复核的自动质量证据。
- 第一章已经进入外部试玩前阶段，后续改动需要更强的回归保护。

### 保留

- 不新增生产依赖；新增依赖均为开发期质量工具。
- 不改变核心玩法定义、内容规模、技术栈或发布方向。
- 不把完整通关流程和 UI 组件覆盖率一口气设为硬门槛。

## v1.4 · 2026-05-20

### 来源

- 用户对照《杀戮尖塔》的遗物节奏后确认：当前法宝给得仍偏多、偏早，第一章改为开局 1 件、第一章过程中 1 件、第一章 Boss 后 1 件，并且每次都是三选一。

### 新增

- 新增第一章法宝三选一节点：开局法宝、途中法宝、Boss 后法宝。
- run 状态新增待选法宝 offer 与已选记录，用于保存三次法宝选择。

### 修改

- 新局不再直接持有 4 件法宝，改为开局从 3 件候选中选 1 件。
- 第一章途中在首个精英节点后发放一次法宝三选一。
- Boss 结算后发放一次法宝三选一，作为后续阶段承接记录。
- 当前第一章商店不再直接出售法宝，商店保留买卡、删牌和临时朱批服务。

### 原因

- 降低首局认知负担，避免玩家一开始被多件牌组外器物同时占据注意力。
- 让法宝更接近“关键阶段奖励”，而不是早期一次性堆满的被动效果列表。

### 保留

- 法宝池规模仍保留 10 件候选定义，满足公开 Demo 内容池目标；变化的是第一章内获得节奏，不是删除法宝系统。
- 法宝仍是牌组外器物，不进入抽牌堆、弃牌堆或普通卡牌奖励池。
- 休整维护反噬预警保留。
- 不新增生产依赖，不改变形、问名、正名、伏诛 / 归册、裁定三类等核心定义。

## v1.3 · 2026-05-19

### 来源

- 用户确认己形方向：己形 72 / 72 起步，跨战保留；休整修形；事件 / 登簿 / 法宝少量增加上限，不随战斗次数自动增加最大形。
- 用户反馈问名源在精英 / Boss 名格数量有限时可能溢出，要求减少大部分纯问名牌。

### 新增

- 新增己形最小闭环：玩家己形进入战斗状态和 run 状态，敌方未封来势扣己形，己形归零进入战斗失败。
- 休整新增修形选项，用于恢复跨战保留的己形。
- 登簿新增少量己形上限成长，和既有香火上限成长并列。

### 修改

- 问名源从 13 个实际入口压到 8 张直接问名 + 1 张天坛间接问名。
- 照骨札、天迹敕、镜契敕、随鞭符从问名源转为抽牌、墨或破形联动牌。

### 原因

- 让敌方来势不只停留在日志压力，而能形成跨战损耗、休整选择和失败条件。
- 避免问名奖励在第一章中后段过度溢出，让问名归册流保留明确取舍。

### 保留

- 不把己形上限做成每场战斗自动成长。
- 不改变敌形、破形、问名、正名、伏诛 / 归册、裁定三类等核心定义。
- 不新增生产依赖。
- 法宝和事件的己形上限收益仍按后续数值调平逐步配置，不在本次扩大为完整新系统。

## v0.6 · 2026-05-17

### 修改

- 技术栈中的 Phaser 版本从 Phaser 3 调整为 Phaser 4.1.x。
- 当前工程锁定 `phaser@4.1.0`，用于 Vite + React 项目的战斗场景渲染层。

### 原因

- 项目是从零开始的新工程，没有 Phaser 3 历史代码包袱。
- 战斗核心逻辑独立在 `src/core`，Phaser 只承载渲染与动效，版本风险被限制在表现层。
- Phaser 4.1.x 对现代 ESM / Vite 工程更友好，并更适合后续卷轴、朱砂、符火、印章、滤镜等 2D 表现需求。

### 保留

- React / Phaser 与核心战斗逻辑继续分离。
- `src/core` 不依赖 React、Phaser 或 DOM。
- 第一阶段仍只做浏览器内核心战斗 MVP，不提前做桌面打包。
