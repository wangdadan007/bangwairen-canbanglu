## v0.7 · 2026-05-18

### 来源

- 基于 `AI开发游戏经验.docx` 的 AI 单机游戏开发经验复盘。
- 结合当前仓库 `AGENTS.md`、PRD、`PROJECT_PLAN.md` 和代码状态做协作护栏优化。

### 新增

- 新增 `docs/tasks/OPEN_QUESTIONS.md`，用于记录 AI 开发过程中不应擅自决定的产品、机制、技术、素材或发布问题。
- 新增 `docs/assets/ASSET_MANIFEST.md`，用于记录第三方素材、AI 生成素材、字体、音频、视频和 contact sheet 批次的来源与授权。
- 在 `PROJECT_PLAN.md` 中补充后续工程优化项：多敌遭遇模型、BattleHud 流程拆分、内容覆盖检查、长流程自动 QA、低成本表现反馈和素材管线。

### 修改

- `AGENTS.md` 不再写死“当前阶段 A”及过期只做 / 暂不做清单，当前真实阶段统一以 `docs/tasks/PROJECT_PLAN.md` 为准。
- `INDEX.md` 与 `PRD-overview.md` 明确阶段表和阶段 A 判断属于启动路线说明，不作为实时进度来源。
- 细化 T40 的 `BattleHud` 拆分标准，并在 T39 / T44 / T48 中补充“不得把新增路线、表现或 QA 状态机重新塞回 BattleHud”的后续约束。
- 复核其余 AI 开发经验补强项，补细 T36 到 T50 的事件 / 卡牌反馈入口、内容覆盖表、多敌兼容、长流程 smoke、素材授权、任务交接记录和候选版 QA 验收。

### 原因

- AI 开发游戏时最容易出现阶段事实漂移、任务边界扩张、上下文丢失、素材授权不清和“能玩但不像游戏”的表现缺口。
- 当前项目已进入阶段 C，如果保留过期阶段口径，后续 AI 容易误判路线、商店、事件、Boss、存档等已完成或可继续扩展的模块。

### 保留

- 当前推荐下一步仍是 T36：事件池与路线中后段补量。
- 核心玩法定义、裁定三类、法宝不入牌堆、封势不能处理异动等红线不变。
- 不新增生产依赖，不引入正式第三方素材。

## v0.6 · 2026-05-17

### 修改

- 技术栈中的 Phaser 版本从 Phaser 3 调整为 Phaser 4.1.x。
- 当前工程锁定 `phaser@4.1.0`，用于 Vite + React 项目的战斗场景渲染层。

### 原因

- 项目是从零开始的新工程，没有 Phaser 3 历史代码包袱。
- 战斗核心逻辑独立在 `src/core`，Phaser 只承载渲染与动效，版本风险被限制在表现层。
- Phaser 4.1.x 对现代 ESM / Vite 工程更友好，并更适合后续卷轴、朱砂、符火、印章、滤镜等 2D 表现需求。

### 保留

- React / Phaser 与核心战斗逻辑继续分离。
- `src/core` 不依赖 React、Phaser 或 DOM。
- 第一阶段仍只做浏览器内核心战斗 MVP，不提前做桌面打包。
