## v1.2 · 2026-05-19

### 来源

- 用户要求建立模板沉淀区，并在每个阶段收尾增加模板化检查项。

### 新增

- 新增 `docs/templates/TEMPLATE_EXTRACTION_NOTES.md`，用于记录可复用游戏模板、工具包和商业化数字产品候选。
- 在 `PROJECT_PLAN.md` 的进度更新规则中新增模板沉淀检查：本阶段是否产生可模板化模块，是否需要记录到模板候选文档。

### 修改

- `AGENTS.md` 的按任务类型读取规则新增模板沉淀 / 商业化抽象入口。
- `AGENTS.md` 的任务收尾格式新增模板沉淀检查项。

### 原因

- 在主游戏开发过程中轻量记录可抽象资产，避免等游戏完成后再回头拆分模板。
- 保持模板化沉淀不干扰当前游戏开发范围。

### 保留

- 不改变当前阶段 D、T51 到 T60 的开发顺序。
- 不新增生产依赖。
- 不抽取或迁移游戏代码。
- 不改变核心玩法定义、题材表达、内容规模、技术栈或发布方向。

- 核心玩法定义、裁定三类、法宝不入牌堆、封势不能处理异动等红线不变。
- 不新增生产依赖，不引入正式第三方素材。

## v0.6 · 2026-05-17

### 修改

- 技术栈中的 Phaser 版本从 Phaser 3 调整为 Phaser 4.1.x。
- 当前工程锁定 `phaser@4.1.0`，用于 Vite + React 项目的战斗场景渲染层。

### 原因

- 项目是从零开始的新工程，没有 Phaser 3 历史代码包袱。
- 战斗核心逻辑独立在 `src/core`，Phaser 只承载渲染与动效，版本风险被限制在表现层。
- Phaser 4.1.x 对现代 ESM / Vite 工程更友好，并更适合后续卷轴、朱砂、符火、印章、滤镜等 2D 表现需求。

### 保留

- React / Phaser 与核心战斗逻辑继续分离。
- `src/core` 不依赖 React、Phaser 或 DOM。
- 第一阶段仍只做浏览器内核心战斗 MVP，不提前做桌面打包。
