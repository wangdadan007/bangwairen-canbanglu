# PRD-checklist：验收标准、QA 与风险清单

版本：v1.4
日期：2026-05-24
建议路径：`docs/prd/PRD-checklist.md`

## 1. 核心战斗 MVP 验收

阶段 A 完成条件：

- `npm install` 成功。
- `npm run dev` 成功。
- `npm run test` 可运行；若失败，必须说明原因。
- 能进入一场测试战斗。
- 玩家有手牌、抽牌堆、弃牌堆、香火、结束回合按钮。
- 敌人有形。
- 至少 5 到 8 张基础卡可用。
- 打出破形牌会正确减少敌人形。
- 敌人形归零后进入胜利状态。
- actionLog 记录抽牌、出牌、费用、破形、胜利。
- 核心逻辑层不依赖 React / Phaser。
- 数据文件新增卡牌或敌人时，不需要改 UI 组件。

阶段 A 可暂不完整实现：路线、商店、法宝、裁定、事件、Boss、桌面打包。

## 2. 问名 / 正名 MVP 验收

- 敌人可配置名格数量。
- 问名牌可以揭示名格。
- 无名目标被问名时触发辨势或弱 fallback，不会废牌。
- 名格全揭示后进入正名。
- 正名触发名破。
- 未正名打空形显示伏诛。
- 正名后打空形显示归册。
- actionLog 记录问名、名格揭示、正名、名破、伏诛 / 归册。

## 3. 来势 / 异动 / 封势验收

- 敌人行动可以配置为来势或异动。
- 来势有数值，可被封势降低。
- 封势不能处理异动。
- 异动必须使用专门逻辑或专门卡处理。
- UI 明确区分来势和异动。
- 普通战行动默认明牌；精英 / Boss、榜裂压力或法宝反噬引发遮势时，不泄露具体行动名、来势值或异动类型。
- 辨势能揭开当前遮蔽行动；若当前行动已明，显示后一动预告。
- 后一动预告在 UI 中必须弱于本次行动，不替代当前行动决策位。
- 至少实现一个异动：偷香或塞污卷。
- 至少实现一个专门应对异动的卡或效果。

## 4. 裁定 MVP 验收

- 归册后能进入裁定页。
- 裁定只出现登簿、朱批、削籍。
- 登簿能写入本局永久规则。
- 普通小怪不做专属登簿；精英和 Boss 归册后的登簿规则必须一怪一条专属。
- 第一章 T66 覆盖窃香吏、火部逃名者、斗部空壳、窃榜使 4 条专属登簿，且 UI、actionLog、结算记录能显示具体登簿名和规则文本。
- 朱批能永久改造一张牌。
- 削籍能给强即时收益并提高榜裂。
- 第一章 T66 削籍提供 4 种判词：裂形符、抄裂得墨、重裂符、预支香墨；每种都必须记录榜裂代价和对应收益。
- 预支香墨只在还有下一场战斗时进入削籍候选；若已经是 Boss 后终局裁定，不展示该选项。
- `刮污成符` 暂不进入第一章削籍池，避免同时删污和加裂形符形成基础削籍的上位奖励。
- 不出现留魂、留名、敌魂卡、Boss 助战。

## 5. 法宝 MVP 验收

- 法宝不是卡牌本体，不进入抽牌堆。
- 法宝显示在独立法宝栏或榜册边缘。
- 法宝有未认主和认主状态。
- 至少 4 件法宝有完整基础效果和认主条件。
- 反噬可以先预留字段，后续再完整实现。
- 实现反噬后，连续两场战斗应作为判定窗口，而不是反噬持续时间。
- UI 显示过载和反噬预警。

## 6. 教学纵切 MVP 验收

阶段 B 完成条件：

- 可连续完成 3 到 5 场战斗。
- 第一场教学：形、破形、问名、正名、伏诛、归册。
- 第二场教学：来势、封势、异动。
- 第三场教学：朱批。
- 奖励池跟随机制解锁。
- 未解锁机制不出现在奖励、商店、事件、法宝描述中。
- 术语首次出现使用“白话 + 术语”。
- 玩法不依赖正式美术也能理解。

## 7. 第一章公开 Demo 验收

阶段 C 完成条件：

- 单局约 30 到 45 分钟。
- 3 个原创可玩角色，均为榜外执簿者分支，不直接使用原典封神人物姓名。
- 每个可玩角色至少有明确的气质差异、初始牌组差异、固定起始法宝差异和 1 条核心机制偏好。
- 首批定稿方向为旧榜执笔“衡简”、照影断名“照微”、莲骨破形“莲烬”；命名和数值可继续打磨，但不得只是换皮。
- 三角色己形上限必须形成明显梯度，初稿为莲烬 78、衡简 72、照微 64，后续只允许基于 QA 调整。
- 照微己形最低，但必须有前期支撑：照骨镜行动信息、较高问名密度、首战教学提示和正名收益路径，不能只是低容错惩罚。
- 三角色版本采用固定角色起始法宝：衡简为打神鞭残节，照微为照骨镜，莲烬为赤绫火轮。第一章途中和 Boss 后仍各进行 1 次法宝三选一。
- 42 到 48 张卡牌。
- 12 个普通敌人单位。
- 18 到 22 个普通遭遇模板。
- 3 个精英敌人。
- 1 个 Boss：窃榜使，含 2 到 3 个行为变体。
- 10 到 12 件法宝。
- 8 到 10 个事件。
- 每条具体路线进入窃榜使 Boss 前应有 8 到 9 场实际战斗；事件、商店和休整作为补给节点，不计入战斗数。
- 路线、商店、休整、结算、设置、存档、继续游戏可用。
- 至少 3 条构筑方向能闭合。
- 玩家能清楚解释问名和归册为什么有收益。

## 8. 平衡目标

| 战斗类型 | 目标回合数 |
|---|---:|
| 普通战 | 3-6 回合 |
| 精英战 | 6-10 回合 |
| Boss 战 | 10-18 回合 |

平衡原则：

- 问名必须有明确收益，但不能让纯破形完全失去意义。
- 纯伏诛路线能通关；伏诛和归册的战后卡牌奖励池一致，均为 3 选 1。
- 归册路线收益更高，但收益差异主要来自裁定、名破和“剥诡计 + 现形反扑”的正名前后阶段变化，而不是卡牌奖励池独占。
- 登簿必须低频、高识别、窄触发：普通归册只走“残名入档”通用模板，每幕段共享额度，当前第一章 Demo 本幕段最多触发 3 次；每次在后续战斗首次正名时消耗 1 层残档并获得 1 墨。同类通用登簿不叠加、不重复抄录；精英 / Boss 才拥有一怪一条专属登簿。登簿不得再通过普通小怪反复叠加香火上限或己形上限。
- T76/T77 后，当前 13 个可正名敌人都应有 `onNamed` / `namedPhase` 数据；至少覆盖遮名失效、偷香 / 污卷降级、召唤 / 劫数交易禁用、斗部挪坛禁用，以及 Boss 按稳行 / 归册 / 高榜裂路线剥掉偷香、遮势或塞污卷压力的测试。
- T79 后，路线验证已枚举所有实际支线，并要求 Boss 前战斗数保持 8 到 9；三角色仍需覆盖所有主线，并额外覆盖非推荐角色支线。
- 卡牌数值必须符合当前章节进度和当前难度曲线，既不能高到碾压同阶段敌人，也不能低到只是无意义弱牌。
- 新增奖励牌应对照同阶段基础牌、敌人形、默认香火和目标回合数做强度判断，优先提供“适度数值提升 + 机制钩子”。
- 第一章前段卡牌不能让玩家稳定跳过正在教学的问名、正名、来势 / 异动处理；后段和精英 / Boss 后奖励再逐步提高效率或复杂度。
- 某一场伏诛不代表玩家放弃问名 / 正名 / 归册构筑；伏诛奖励仍应能出现问名、正名、朱批、三坛或破形方向牌。
- 封势不能万能，不能处理异动。
- 异动反制牌不能过窄到完全卡手，必须有 fallback。
- 朱批不能只加数值，要制造构筑变化；强词条不能贴到 0 费抽牌、0 费回香或已经自带抽牌 / 返香火的循环件上。
- 第一章朱批池目标为 12 种效果，每次朱批服务按解锁机制、目标牌兼容性、循环风险、角色倾向和路线倾向展示 4 个可触发候选；不能把三坛、墨、劫数等未解锁机制提前刷给玩家。
- 削籍收益必须诱人，但榜裂代价必须能被感知。
- T78 后削籍增强只走短效交付：裂形符 / 重裂形符下一场开局置手，预支香墨下一场开局 1 墨且首回合香火 +2，抄裂得墨保持立即墨资源救急；不做长期费用上限、己形上限、稳定抽牌引擎或免费净化服务。
- 削籍变体之间不能出现明显完胜关系：净化、删负面、双收益等强效果应放到事件 / 休整 / 法宝服务另行定价，不直接并入基础削籍池。
- 当前设计判断：归册期望收益应高于伏诛，因为玩家为问名、正名、现形反扑和战斗拖延支付了成本；但 T78 暂不继续抬高归册本身，优先通过削籍短效增强让裁定三选内部更平衡。

卡牌数值验收：

- 每张新增或调整过的卡牌，都应能说明所属章节 / 解锁阶段、参照基础牌和预期用途。
- 对当前 MVP 数据，可以用数据测试约束早期奖励牌的临时上限；若敌人形、香火经济或章节曲线变化，必须同步调整测试阈值和说明。
- 若某张牌让同阶段普通战稳定低于目标回合数，或让问名 / 正名 / 封势 / 断异动机制失去意义，应降低数值、增加条件、提高费用或延后解锁。

## 9. PC / Steam 准备验收

公开 Demo 前至少满足：

- 16:9 分辨率下表现良好，优先 1920x1080。
- 支持窗口化、全屏、无边框窗口。
- 支持常用分辨率缩放。
- 支持鼠标操作。
- 后续预留手柄适配。
- 支持设置音量、窗口模式、语言。
- 支持本地存档。
- 支持崩溃日志或错误日志。
- 支持关闭动画 / 降低特效。
- 支持中文，预留英文 localization。
- 不把中文文案硬编码在逻辑里。
- 不在构建中包含开发密钥、账号、API token。
- 所有第三方素材记录来源、授权、用途、是否可商用。

## 10. 风险清单

| 风险 | 等级 | 处理方式 |
|---|---|---|
| 术语过多，新手理解困难 | 高 | 首次白话 + 术语、tooltip、逐战斗解锁 |
| 问名收益不足，玩家只破形 | 高 | 名破、归册奖励、裁定收益必须明确 |
| 问名收益过强，破形直攻失效 | 中 | 伏诛路线可通关，名破有敌人类型系数 |
| 封势变成万能防御 | 高 | 代码层禁止封势处理异动 |
| 法宝反噬让玩家觉得负收益 | 中 | 先做认主正反馈，再做反噬；反噬必须预警 |
| 第一章怪物池太薄 | 中 | 12 普通敌人 + 18 到 22 遭遇模板 |
| 三角色扩大平衡和 QA 成本 | 高 | 先定角色机制边界，再做最小初始牌组 / 起始法宝差异；三角色都必须跑通第一章基础验证 |
| 开发一开始范围过大 | 高 | 先核心战斗 MVP，不直接做完整第一章 |
| 美术漂向二次元封神手游 | 高 | 残榜、朱砂、旧庙、青铜、木刻基线 |
| 与竞品观感撞车 | 中 | 避免“攻击/格挡”语言、普通爬塔地图、神将立绘收集 |
| 授权不清 | 高 | 第三方素材必须记录授权，不清晰不得进正式构建 |

## 11. 当前不可做清单

当前阶段不要做：

- 完整正式版全部章节。
- 超出首批 3 个原创执簿者的大规模多角色系统。
- 多 Boss 随机池。
- 完整 Steam 页面素材。
- 复杂法宝反噬动画。
- 复杂事件链。
- 敌魂卡。
- Boss 饶恕助战。
- 留魂 / 留名。
- 大量原典封神人物可玩立绘。
