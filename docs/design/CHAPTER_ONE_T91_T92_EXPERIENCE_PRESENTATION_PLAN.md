# T91-T92 第一章体验与表现推进方案

日期：2026-05-25

## 1. 结论

T90 之后，第一个 Boss 前不再优先增加新系统。当前更高价值的优化是两件事：

1. T91：补首局理解和 Boss 前准备感，让玩家知道自己为什么问名、为什么归册、现在缺什么、进 Boss 前是否准备好了。
2. T92：做战斗美术与动画垂直切片，让核心操作有可感知反馈，并让第一章气质从“系统接上了”进入“能被玩家记住”的状态。

第三类“角色差异进一步深化”暂不做完整角色专属牌池、专属事件链或新机制。T91/T92 只允许通过角色选择页、起始法宝、奖励倾向解释和战斗反馈强化角色感知。若 T92 后外部试玩仍反馈三角色像换皮，再单独开后续任务。

## 2. T91：首局理解与 Boss 前准备感

目标：在不新增卡牌、敌人、Boss、章节和核心机制的前提下，验证并优化玩家是否能从现有 UI 理解第一章的关键取舍。

### 2.1 需要验证的问题

- 玩家是否能在前三场理解形、问名、正名、伏诛、归册、裁定的关系。
- 玩家是否能理解伏诛和归册卡牌奖励同池，归册收益主要来自裁定、名破和剥诡计。
- 玩家是否能理解香封是一次性战术资源，应该留给精英 / Boss 或高压战斗。
- 玩家是否能在路线、商店、休整和 Boss 前节点判断自己缺收束、防守、净卷、护名、问名还是资源。
- 玩家是否能看出法宝 offer、朱批、删牌和香封购买是在塑造本局构筑，而不是孤立服务。

### 2.2 实现方向

- 新增或更新一份 T91 QA 记录，至少覆盖一个首局新手视角流程和两条 Boss 前准备流程。
- 复核 `firstRunGuidance`，只补状态化短提示，不做长教程墙。
- 在 Boss 前或临近 Boss 的路线 / 休整 / 商店节点增加“临战整册”式摘要：己形、墨、香火钱、香封、法宝、朱批、榜裂 / 劫数和当前主要风险。
- 摘要只解释准备状态，不给自动推荐出牌，不暴露内部权重算法。
- 在商店 / 休整页补更明确的不可操作原因和取舍提示，例如删牌、净卷、护名、香封槽满、法宝维护。
- 若真实流程暴露阻断级体验问题，可以在 T91 内做小幅 UI / 文案 / 节点反馈修正；数值、敌人压力或奖励权重只在证据明确时小调。

### 2.3 不做内容

- 不新增卡牌、敌人、遭遇模板、香封、法宝、Boss、章节或角色。
- 不新增生产依赖、正式素材或音频。
- 不做完整随机地图、Ascension / Daily / Custom、局外数值成长或外层挑战模式。
- 不做角色专属牌池、角色专属事件链或第四角色。
- 不重写教程系统，不把 UI 变成长篇说明书。

### 2.4 验收标准

- T91 QA 文档记录首局理解、Boss 前准备感、商店 / 休整取舍和剩余问题。
- Boss 前准备摘要能让玩家看见当前资源、构筑锚点和风险状态。
- UI 文案继续使用项目术语，并在首次或关键处保留白话解释。
- `npm run typecheck`、`npm run build` 和相关测试通过；若改动只涉及文档，则至少运行 `git diff --check`。
- 若涉及浏览器页面，需做一次本地浏览器冒烟并记录结果。

## 3. T92：战斗美术与动画垂直切片一期

目标：在 T91 确认第一章 Boss 前体验没有明显阻断后，进入低风险表现制作。T92 不是正式美术量产，而是可玩切片的视觉和动效基准。

### 3.1 覆盖范围

- 战斗桌案基底：残榜、朱砂、旧庙、青铜、木刻、香灰、暗金火光等视觉关键词。
- 卡牌框和状态标签：让普通、精良、秘传、临诏、香封、朱批状态更容易扫读。
- 敌人占位剪影或低成本图像：优先覆盖纸面鬼、偷香鼠、无灯庙祝、三名精英和窃榜使。
- 核心反馈：破形、问名、正名、伏诛、归册、登簿、朱批、削籍、封势、异动、遮势、法宝认主、法宝反噬和 Boss 终审压力。
- 角色可见差异：只做标题页 / 角色选择 / 起始法宝 / 战斗反馈的气质强化，不做角色新系统。

### 3.2 资产与授权边界

- 优先使用自制低成本素材、程序化纹理、CSS / Phaser 表现和已登记的候选资产工作流。
- 如使用 AI 生成图像或第三方素材，必须记录来源、授权、用途和是否可商用到资产清单；授权不清不得进入正式构建。
- 不新增字体、音频、美术素材包或其他生产依赖，除非用户单独确认。

### 3.3 动画边界

- 动画必须服务可读反馈，不追求复杂演出。
- 所有关键动画都要遵守设置页动画开关和 `prefers-reduced-motion` 兜底。
- 表现层只消费核心状态和 actionLog，不把规则判定写进 React 或 Phaser 表现层。
- Boss 只做终审压迫感和关键状态反馈，不做完整剧情演出。

### 3.4 不做内容

- 不量产全套正式美术资产。
- 不做 Steam 商店页素材、宣传图、完整角色立绘和正式过场。
- 不做正式音乐或完整音效库。
- 不新增生产依赖，不改技术栈。
- 不新增玩法系统，不借表现任务改卡牌、敌人、经济或路线数值。

### 3.5 验收标准

- 浏览器内第一章代表性切片可见：前三场教学、中段多敌、一名精英和窃榜使 Boss 的关键反馈。
- 1920x1080、1366x768、1280x720 至少三档视口无明显文本重叠、按钮溢出或核心信息遮挡。
- 动画开关关闭后，核心反馈仍可通过静态状态、日志或短提示理解。
- 若触及 Phaser 画面，需确认画面非空、关键对象正确显示、交互不被遮挡。
- `npm run typecheck`、`npm run build`、相关测试和浏览器冒烟通过；如需完整收口再跑 `npm run verify`。

## 4. T92 后闸门

T92 完成后再判断下一步：

- 如果玩家仍看不懂问名 / 归册 / Boss 前准备，回到 T91 类体验修补。
- 如果玩家觉得战斗反馈仍弱，继续 T93 做表现二期，而不是加新系统。
- 如果三角色仍被认为差异不足，再单独设计角色签名牌 / 专属事件 / 专属法宝反馈，不并入 T91/T92。
- 如果第一章已经能被小范围玩家完整理解，再进入外部试玩反馈修复或 Steam Demo 发布准备。
