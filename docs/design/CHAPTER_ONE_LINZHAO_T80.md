# T80 临诏机制设计与第一章试点方案

版本：v0.2
日期：2026-05-24
状态：已完成最小开发落地
建议路径：`docs/design/CHAPTER_ONE_LINZHAO_T80.md`

## 1. 设计结论

第一章可以加入一个与其他牌组构筑肉鸽中“战斗内持续引擎”功能等价的系统，但不直接称为能力牌，也不照搬通用攻击 / 防御增益。

本项目采用“临诏”作为封神榜化表达：

> 临诏是一种可进入牌组的战斗内持续规则牌。打出后不进弃牌堆，而是挂入战斗页的“临诏区”；本场战斗持续生效，战后消失。

临诏的核心价值是让玩家在长战中先投入香火、墨或出牌节奏，随后围绕问名、正名、封势、断异动、朱批等项目独有机制获得持续回报。

## 2. 系统边界

| 系统 | 定位 | 持续范围 |
|---|---|---|
| 临诏 | 由牌打出的本场战斗持续规则 | 本场战斗 |
| 登簿 | 归册裁定后写入榜格的本局永久规则 | 本局 |
| 朱批 | 对已有卡牌添加永久改造词条 | 本局，绑定单张牌 |
| 法宝 | 牌组外器物，可触发效果或生成临时牌 | 本局，牌组外 |
| 三坛 | 有明确触发窗口的一次性战斗时间轴状态 | 单次触发窗口 |

临诏不是登簿，不写入本局永久榜格；不是朱批，不永久改造某张牌；不是法宝，不能作为器物常驻牌组外；不是三坛，不只在一个时间窗口触发后归寂。

临诏牌可以被朱批，但朱批只写在“牌本体”上：该牌打出时先落入临诏区，再触发这张牌自身携带的朱批词条。已经挂入临诏区的临诏状态不是卡牌，不可再作为朱批目标；朱批候选仍按循环风险过滤，返香火等强循环词条不贴到临诏牌上。

## 3. 第一版规则

- 临诏牌进入普通牌组和奖励 / 商店 / 事件渠道，但第一版优先通过改造现有 48 张卡牌中的 2 到 3 张敕令或支援牌试点，不扩大第一章卡牌上限。
- 打出临诏牌后，该牌移出本场抽牌 / 弃牌循环，挂入“临诏区”；战斗结束后恢复为牌组中的普通卡牌定义，不产生跨战斗常驻状态。
- 临诏区上限采用 2 到 3 的可调参数；第一版默认 2，若 Boss / 精英长战铺设空间不足，再调到 3。
- 同名临诏不叠加。若临诏区已满，首版不做自动顶替；玩家不能打出新的临诏，除非牌面明确写有替换规则。
- 每次临诏打出、入区、触发、因上限失败和战斗结束清除，都必须写入 actionLog。
- 临诏触发条件必须围绕本项目独特机制：问名、正名、名破、封势、断异动、朱批、墨、榜裂、归册。避免做成通用“本场破形 +1”或“每回合获得封势”。
- T81 调平后，明烛临诏和朱痕临诏后移到 `stage_human_altar`，不进入前三场教学奖励或商店窗口；压煞临诏继续作为更后段的封势 / 断异动试点。

## 4. 第一批试点方向

| 临诏候选 | 费用建议 | 方向 | 规则草案 | 设计目的 |
|---|---:|---|---|---|
| 明烛临诏 | 1 香火 | 问名 / 正名 | 本场战斗中，每回合第一次问名后，本回合下一次破形 +1；若目标已正名，改为 +2。 | 让问名牌能铺成长战节奏，而不是只为归册支付成本。 |
| 压煞临诏 | 1 香火 | 封势 / 断异动 | 本场战斗中，每回合第一次完全封住来势或成功断异动后，对当前目标破形 2；若目标已正名，改为破形 3。 | 把防守回合转成收束压力，同时保持封势和异动分工。 |
| 朱痕临诏 | 1 香火 | 朱批 / 改牌引擎 | 本场战斗中，第一次触发朱批词条后抽 1 张并获得 1 香火；之后每回合第一次触发朱批时获得 1 香火。 | 给朱批改牌流一个战斗内铺设目标，但不提供无限循环。 |

以上三张已在 T80 最小开发中写入 `cards.json` 和 battle resolver。后续仍可按完整局测试结果调整数值、费用和解锁阶段。

## 5. 候选改造牌位

第一版不新增卡牌数量，优先从现有敕令 / 支援牌中改造 2 到 3 张：

| 现有牌位 | 当前定位 | 可试点临诏方向 |
|---|---|---|
| `card_trace_name_slip` | 已改为明烛临诏 | 问名后准备本回合下一次破形加成。 |
| `card_name_net_talisman` | 已改为压煞临诏 | 完全封势或成功断异动后转破形。 |
| `card_red_ink_trial` | 已改为朱痕临诏 | 触发朱批词条后返香火 / 抽牌。 |

如果开发时发现上述牌位承担了重要可达性或教学职责，应优先改造其他同阶段敕令 / 支援牌，而不是直接把卡牌池扩到 50 张以上。

## 6. UI 与日志需求

- 战斗页增加“临诏区”，位置建议靠近残榜 / 法宝侧栏，但视觉上要和法宝、三坛区分。
- 每个临诏显示名称、来源牌、剩余战斗内持续状态和本回合是否已触发。
- tooltip 说明触发条件、同名不叠加和战后消失。
- actionLog 至少覆盖：打出临诏、入区成功、同名被拒、临诏区已满、触发成功、触发跳过和战斗结束清除。
- 首局引导只在第一次获得或打出临诏时短提示，不提前在前三场教学战讲解。

## 7. 不做内容

- 不叫“能力牌”。
- 不把临诏做成登簿、法宝或朱批的替代系统。
- 不新增生产依赖、正式素材、音频或新 UI 框架。
- 不扩大第一章公开 Demo 卡牌上限。
- 不加入通用攻击力、通用防御力或每回合固定收益模板。
- 不让临诏绕过封势 / 异动边界；封势仍只处理来势，异动仍需断异动、地坛、专门牌或法宝处理。

## 8. T80 验收建议

T80 最小开发已完成：

- 战斗状态能记录当前激活临诏。
- 2 到 3 张现有牌改造成临诏牌，并保持卡池总数不超过 48。
- 临诏打出后从本场抽弃牌循环移入临诏区，战斗结束后不留下战斗内状态。
- 同名不叠加，上限满时有明确不可打出原因。
- 明烛 / 压煞 / 朱痕三类覆盖问名、封势 / 断异动、朱批三个方向。
- 临诏牌可被朱批；朱批只增强卡牌本体，不增强已在临诏区的状态。
- actionLog 和 HUD 能解释临诏打出、触发和未触发原因。
- 相关纯逻辑测试覆盖入区、触发、上限、同名拒绝和战斗结束清除。
